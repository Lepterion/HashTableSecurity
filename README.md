# Secure Hash Table (SHT) & In-Memory Vault 🛡️

## Опис проєкту
Кастомна реалізація хеш-таблиці «ключ-значення» з акцентом на захист даних в оперативній пам'яті (In-Memory Protection).

Стандартні структури даних (зокрема `std::unordered_map`) зберігають інформацію у відкритому вигляді (plaintext). Це робить чутливі дані вразливими до **неприцільного** сканування пам'яті (Memory Dumping / RAM Scanning) дебагерами чи шкідливим ПЗ. Наше рішення тримає всі значення **зашифрованими** і розшифровує їх лише на час обробки запиту.

## 🎯 Threat Model (Модель загроз)
Зловмисник сканує адресний простір процесу (наприклад, Cheat Engine) у пошуках паролів чи токенів. **Мета:** зробити так, щоб дамп пам'яті повертав лише зашифроване «сміття» замість реальних значень, зберігши швидкість доступу $O(1)$ у середньому.

> **Чесне обмеження.** In-memory захист без апаратних анклавів не рятує від *прицільної* атаки: той, хто зчитав увесь адресний простір, теоретично знайде і ключ сесії, тож зможе дешифрувати дані. Шифрування ефективне саме проти **неприцільного** скану — у пам'яті не лишається відкритих рядків і повторюваних патернів, а ключ затирається при завершенні сесії.

## 🔒 Від чого захищає, а від чого ні

Ключове розрізнення — чи має зловмисник **лише читання сирих байтів** пам'яті, чи ще й **виконання коду** всередині нашого процесу. Шифрування "at-rest" допомагає проти першого і майже не допомагає проти другого.

### ✅ Захищає (пасивне / неприцільне читання пам'яті)
* **Grep по дампу пам'яті.** Зловмисник чи зловмисне ПЗ шукає в RAM відомі патерни (рядки `password=`, формати API-ключів, номери карток за регуляркою). Наші значення лежать як випадкові байти й під жоден патерн не підпадають.
* **Випадкові витоки пам'яті назовні.** Core dump після краху, файл гібернації (`hiberfil.sys`), своп-розділ, crash-репорт у телеметрії. Якщо такий артефакт витече — у ньому немає відкритих секретів.
* **Сканер з іншого процесу без привілеїв.** Інструмент типу Cheat Engine, що читає пам'ять цільового процесу: бачить лише «сміття».
* **Скорочення вікна відкритого тексту.** Плейнтекст існує лише на час одного `find()` у тимчасовому буфері, який одразу затирається; у самій таблиці його немає ніколи.

### ❌ НЕ захищає (активний зловмисник усередині процесу)
* **Виконання коду в нашому процесі.** Якщо чужий код працює всередині (вставлена бібліотека, налагоджувач через `ptrace`, зловмисний плагін) — він знайде ключ сесії в тій самій пам'яті, або просто викличе `decrypt()` / зчитає плейнтекст під час `find()`. Ключ і дані живуть в одному адресному просторі — це фундаментальна межа будь-якого суто програмного in-memory захисту.
* **Плейнтекст на «кінцях».** Значення, яке ви передали в `insert()`, і рядок, що повернув `find()`, — це звичайні `std::string` у вашому коді; вони не зашифровані. Захищене лише *сховище*, а не вся програма.
* **Прицільна атака на ключ.** Хто знає структуру `SecurityProtocol`, той локалізує 32 байти ключа і дешифрує все. Без апаратної підтримки (TPM, SGX/TEE, апаратне шифрування RAM) інакше бути не може.
* **Цілісність і підробка.** ChaCha20 дає лише конфіденційність. Зловмисник може *змінити* шифротекст (хоч і не знаючи, на що) — від цього потрібен MAC (наприклад, ChaCha20-Poly1305).
* **Cold-boot / DMA / side-channel атаки** на саму мікросхему пам'яті.

### 💡 Приклади та межі застосування

* **Змінні в інтерпретаторі (Python `dict`).** Так, CPython тримає простори імен (`globals()`, атрибути об'єктів) у словниках — це хеш-таблиці. Але замінити їх на шифрувальну таблицю **недоцільно**: інтерпретатору щомиті потрібен плейнтекст, тож дешифрування йшло б постійно, а ключ усе одно в тому ж процесі — виграшу нуль, втрата швидкості величезна. Реалістично шифрувати **окремі секрети** (токени, паролі, ключі сесій), а не всі змінні: тримати їх у такому сховищі й діставати точково.
* **Supply-chain атака (зловмисний пакет).** Тут потрібно розрізняти два випадки:
  * Якщо пакет лише **пасивно фотографує пам'ять** (наприклад, дампить власний процес і шле дамп назовні) — шифрування **допомагає**: у дампі замість секретів випадкові байти.
  * Якщо ж пакет **виконує код у вашому процесі** (а саме так працює імпортований залежний пакет) — він може зчитати ключ, перехопити (`hook`) функції доступу або викликати API сховища. Тут захист **майже не діє**. Тобто SecureHashTable піднімає планку проти «лінивого» зловмисного коду, що просто грепає пам'ять, але не зупиняє цілеспрямований код із виконанням у процесі.
* **Де це справді доречно:** менеджер секретів / vault у застосунку, кеш токенів, тимчасове зберігання ключів шифрування — усе, що рідко читається, але не має «світитися» у дампах та логах. Для серйозних вимог це доповнюють `mlock`/`VirtualLock` (заборона свопу), апаратними анклавами та автентифікованим шифруванням.

## ⚙️ Архітектура та Протокол захисту

1. **Справжнє шифрування — ChaCha20 (RFC 8439).** Потоковий шифр реалізовано з нуля, без зовнішніх бібліотек. Коректність підтверджено офіційним тест-вектором RFC 8439 (збіг байт-у-байт).
2. **Динамічний 256-бітний ключ сесії.** Генерується випадково при створенні таблиці із системного джерела ентропії (`std::random_device`). Між запусками шифротекст відрізняється.
3. **Унікальний нонс на кожне значення.** Однакові секрети дають різний шифротекст — немає повторюваних патернів для аналізу.
4. **Шифрування «at rest».** У таблиці значення завжди зашифровані; `find()` дешифрує лише в тимчасовий буфер, який одразу затирається. Збережена копія лишається зашифрованою.
5. **Secure Wiping.** При видаленні вузла та при знищенні таблиці пам'ять примусово затирається нулями через `volatile`-вказівник (щоб компілятор не викинув цикл під оптимізацією). Ключ сесії затирається в деструкторі.

## 🗂️ Структура репозиторію
* `protocol.cpp` — Ядро: реалізація ChaCha20, `SecureNode`, `SecurityProtocol` (encrypt/decrypt/wipe) та **`SecureHashTable`** — справжня хеш-таблиця з ланцюжковим вирішенням колізій (`insert` / `find` / `erase`).
* `test.cpp` — Автоматизовані тести: коректність шифрування/дешифрування, унікальність нонсів, відсутність відкритого тексту в пам'яті, безпечне затирання, а також операції хеш-таблиці (вставка, оновлення, пошук, видалення, 100 ключів із колізіями).
* `benchmark.cpp` — Бенчмарки: порівняння швидкості `insert`/`find` із plaintext `std::unordered_map` та вимір пропускної здатності ChaCha20 (МБ/с).
* `test_app.cpp` — Міні-застосунок «Secure Vault»: наочно показує різницю між тим, що бачить хакер (Raw RAM Dump), і тим, що отримує легальний користувач через API таблиці.
* `protocol_openssl.cpp` — Альтернативна версія протоколу (простір імен `ossl`) на основі **OpenSSL** (`EVP_chacha20`) замість власної реалізації шифру. API ідентичний.
* `benchmark_openssl.cpp` — Порівняння трьох реалізацій: ручний ChaCha20 vs OpenSSL vs plaintext.

## 🚀 Як запустити

### Через CMake (рекомендовано)
```bash
cmake -S . -B build
cmake --build build -j
ctest --test-dir build --output-on-failure   # запустити тести

# Виконувані файли у build/
./build/app                 # демо Secure Vault
./build/benchmark           # бенчмарк ручної реалізації
./build/benchmark_openssl   # порівняння з OpenSSL (якщо libcrypto знайдено)
```
> Ціль `benchmark_openssl` додається автоматично лише за наявності OpenSSL; інакше CMake її пропускає.

### Вручну через g++
Ядро компілюється стандартом C++17 без зовнішніх залежностей:
```bash
g++ -std=c++17 -O2 test.cpp -o test_runner && ./test_runner
g++ -std=c++17 -O2 test_app.cpp -o app && ./app
g++ -std=c++17 -O2 benchmark.cpp -o benchmark && ./benchmark
g++ -std=c++17 -O2 benchmark_openssl.cpp -o benchmark_openssl -lcrypto && ./benchmark_openssl
```

## 📋 API хеш-таблиці
```cpp
SecureHashTable vault(16);             // 16 бакетів, власна сесія шифрування
vault.insert("key", "secret value");   // шифрує і зберігає
std::string out;
if (vault.find("key", out)) { ... }     // дешифрує у out
vault.erase("key");                     // затирає та видаляє
```

## 📊 Результати бенчмарку
Заміри на 100 000 записів (`-O2`; абсолютні цифри залежать від машини):

| Операція | `unordered_map` (plaintext) | `SecureHashTable` (ChaCha20) | Ціна безпеки |
|---|---|---|---|
| insert | ~17 ms | ~29 ms | ~1.7× |
| find   | ~7 ms  | ~24 ms | ~3.4× |

Пропускна здатність ChaCha20: **~480 МБ/с** (шифрування 64 МБ).

**Висновок:** шифрування «на льоту» додає помірний оверхед (find ~3–4×, insert ~2×) при повній відсутності відкритих значень у пам'яті. Для зберігання секретів (паролі, токени, API-ключі), де операцій небагато, ця ціна цілком прийнятна.

## 🔬 Порівняння з OpenSSL
`benchmark_openssl.cpp` зіставляє власну реалізацію ChaCha20 з промисловою з OpenSSL. Результати неочевидні й залежать від сценарію:

| Сценарій | Ручний ChaCha20 | OpenSSL (`EVP_chacha20`) |
|---|---|---|
| Масовий потік (256 МБ одним буфером) | ~570 МБ/с | **~3300 МБ/с** (≈5.8× швидше) |
| Таблиця, 100k дрібних значень (find) | **~23 ms** | ~48 ms (≈2× повільніше) |

**Чому так:** на великих даних OpenSSL виграє завдяки векторизації (SIMD/AVX) та асемблерним вставкам. Але на безлічі **дрібних** операцій його перемагає накладна витрата: на кожен `encrypt`/`decrypt` створюється і знищується `EVP_CIPHER_CTX`, і це налаштування домінує над корисною роботою для значень у ~30 байтів. Легка ручна реалізація такого setup-коштy не має.

> Накладну витрату OpenSSL можна зменшити, перевикористовуючи один `EVP_CIPHER_CTX` (через `EVP_CIPHER_CTX_reset`) замість створення нового на кожен виклик — тут навмисно лишено простий варіант для наочності.
>
> **Практичний висновок:** для секрет-сховища (мало даних, багато коротких звернень) достатньо й навіть вигідніше легке власне рішення; для шифрування великих обсягів — однозначно бібліотека.
