# Secure Hash Table (SHT) 🛡️

## Опис проєкту
Secure Hash Table — це кастомна реалізація хеш-таблиці, спроєктована з акцентом на безпеку даних в оперативній пам'яті (In-Memory Protection). 

Стандартні структури даних зберігають інформацію у відкритому вигляді (plaintext), що робить її вразливою до сканування пам'яті (Memory Dumping / RAM Scanning). Наше рішення інкапсулює дані, обфускуючи їх під час зберігання, щоб захистити чутливу інформацію (наприклад, ігрові стани, паролі, ключі) від стороннього втручання.

## 🎯 Проблема, яку ми вирішуємо (Threat Model)
Якщо зловмисник або програма-сканер (наприклад, Cheat Engine) отримує доступ до адресного простору нашого процесу, він може легко знайти та змінити змінні. 
Наша мета: зробити так, щоб дамп пам'яті повертав лише "сміття" (зашифровані байти) замість реальних значень.

## ⚙️ Архітектура та Протокол захисту
*(Цей розділ ти заповниш детальніше, коли колега закінчить роботу над протоколом, але базова логіка така:)*

1. Генерація ключа: При ініціалізації хеш-таблиці генерується унікальний динамічний ключ SessionKey.
2. Запис (Insert): При додаванні пари (Key, Value), значення обробляється алгоритмом обфускації (наприклад, побітовий XOR з `SessionKey`) перед записом у бакет.
3. Читання (Get): При запиті значення по ключу, дані витягуються з бакета, розшифровуються в регістрі процесора і повертаються користувачу. В самій RAM дані завжди залишаються зашифрованими.

## 🛠️ API (Інтерфейс використання)

Структура підтримує стандартний набір операцій, але приховує складність безпеки під капотом:

* insert(key, value) — хешує ключ, шифрує значення, зберігає в пам'яті.
* get(key) — знаходить бакет, дешифрує значення "на льоту" і повертає його.
* remove(key) — безпечно видаляє вузол (можливо, з перезаписом пам'яті нулями перед видаленням вказівника, щоб уникнути витоку даних — *Data Wiping*).

## 🧪 Тестування (Security Verification)
Наш тестовий набір включає:
1. Функціональні тести: Перевірка колізій, ресайзингу таблиці та коректності CRUD операцій.
2. Memory Scan Test: Спеціальний тест, який отримує прямий вказівник на "сиру" пам'ять таблиці та намагається знайти там оригінальний рядок або число. Тест проходить успішно лише тоді, коли пошук видає помилку (дані надійно приховані).
